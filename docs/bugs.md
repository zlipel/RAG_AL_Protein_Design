## Bug #1 — CRITICAL: `batch_y` extraction is wrong

**Status:** ✅ Fixed — `commit 407fcf8` (merged to `audit/agent-scaffold`)

**File:** `src/rag_al/loop/runner.py`, line 154

**Code:**
```python
batch_y = dataset.labeled_y[-len(selected_local):]
```

**Problem:**
`labeled_y` is produced by indexing `__fitness` with the boolean `_labeled_mask`
over the full dataset in original row order. After `reveal()`, the newly labeled
variants are inserted at their positions in the dataset order — they are NOT
appended to the end of the array.

Example: 100 variants; initial labeled set at rows [10, 20, 30 ...]. We select
pool variants at global rows [2, 7]. After reveal, labeled_y is indexed in row
order: rows [2, 7, 10, 20, 30 ...]. The slice `[-2:]` returns rows [20, 30] —
the wrong variants. The batch fitness metrics (`batch_mean_fitness`) are silently
wrong every round.

**Fix applied:**
Save `global_selected` before `reveal()`, then read batch fitness via
`dataset.fitness_at(global_selected)`.

## Bug #2 — MEDIUM: `dataset._df` accessed directly from runner

**Status:** ✅ Fixed — `commit 407fcf8` (merged to `audit/agent-scaffold`)

**File:** `src/rag_al/loop/runner.py`, lines 84 and 148–149

**Code:**
```python
wt_sequence: str = dataset._df["wt_sequence"].iloc[0]
batch_sequences = list(dataset._df.loc[global_selected, "mutated_sequence"])
```

**Problem:**
`_df` is a single-underscore (semi-private) attribute that contains the full
DataFrame — including the `fitness` column. The runner currently only reads
`wt_sequence` and `mutated_sequence`, so no fitness leaks today. But this is
fragile: any future edit that reads `dataset._df["fitness"]` would silently
succeed and leak all labels.

**Fix applied:**
Added `wt_sequence` property and `get_sequences()` / `get_variant_ids()` to
`ALDataset`. Runner now uses those public helpers exclusively.

## Bug #3 — MEDIUM: Self-label inclusion in `RetrievalAugmentedEncoder`

**Status:** ✅ Fixed — `commit 1a95adf` (merged to `audit/agent-scaffold`)

**File:** `src/rag_al/representations/retrieval.py`, lines 86–87

**Code:**
```python
dists, _ = self._knn.kneighbors(self._labeled_embeddings)
self._dist_scale = max(float(np.median(dists)), 1e-8)
```
and in `_retrieval_features()`:
```python
dists, idxs = self._knn.kneighbors(query_embeddings, n_neighbors=k)
```

**Problem:**
When `runner.py` calls `encoder.transform(df_labeled)` to get `X_labeled` for
surrogate training, `transform()` queries the kNN index with the same points
that were used to build it. sklearn's `NearestNeighbors.kneighbors()` returns
each training point as its own nearest neighbor (distance = 0).

This means the retrieval features for each labeled variant include its **own**
fitness score in `mean_y`, `max_y`, and `std_y`. The surrogate is then trained
on features that directly encode self-fitness for every training point.

Pool variants are clean — they are not in the kNN index. This is not a
pool→labeled leakage, but it creates an asymmetry: the surrogate learns
from self-informative labeled features but must predict pool variants whose
retrieval features do not include self-fitness. This can inflate surrogate
training accuracy and distort calibration.

**Fix applied:**
Added `transform_labeled()` override to `RetrievalAugmentedEncoder` that fetches
`k+1` neighbors and discards column 0 (self). `_dist_scale` in `fit()` likewise
excludes the self column. `runner.py` calls `encoder.transform_labeled(labeled_df)`
for the labeled set and `encoder.transform(pool_df)` for the pool.

## Bug #4 — MINOR: Dead code in `physicochemical.py`

**Status:** ✅ Fixed — `commit 1a95adf` (merged to `audit/agent-scaffold`)

**File:** `src/rag_al/representations/physicochemical.py`, inside `_encode_one()`

**Code:**
```python
net_charge += _AA_HYDROPATHY.get(aa, 0.0) * 0.0  # placeholder replaced below
```

**Problem:**
`_AA_HYDROPATHY.get(aa, 0.0) * 0.0` is always `0.0`. This line adds nothing
to `net_charge` and is dead code. The comment "placeholder replaced below" is
misleading.

**Fix applied:**
Deleted the line entirely. No behavior change.

## Gap #1 — Selected variant indices not saved

**Status:** ✅ Fixed — `commit 407fcf8` (merged to `audit/agent-scaffold`)

**File:** `src/rag_al/cli/benchmark.py` / `src/rag_al/loop/runner.py`

**Problem:**
The results CSV recorded per-round metrics but not which specific variants were
selected each round. Experiments could not be fully reconstructed and the
acquisition sequence could not be audited post-hoc.

**Fix applied:**
`run_al_loop()` now returns `(results_df, selections_df)`. `benchmark.py` saves
`seed_<N>_selections.csv` alongside `seed_<N>.csv`. Selections columns:
`round, global_index, variant_id, fitness`.

## Performance Issue — ESMEncoder cache misses every round

**Status:** ✅ Fixed — `commits a657b40, e661ae1` (on `fix/plm-cache-hashmap`,
pending merge to `audit/agent-scaffold`)

**File:** `src/rag_al/representations/plm.py`, `_try_load_cache()`

**Problem:**
The cache was keyed by the exact ordered list of `variant_id` values passed to
`transform()`. During the AL loop, `transform()` is called separately on
`labeled_df` and `pool_df`. Neither of these matched the full-dataset list used
when saving the cache. Result: cache miss every call, embeddings recomputed from
scratch each round. Additionally, `transform()` contained a bare `for` (syntax
error) from an incomplete earlier patch, crashing any cache-hit path.

**Fix applied:**
Replaced the two-file array cache (`embeddings_*.npy` + `variant_ids_*.npy`) with
a single `{sha256(sequence) → embedding}` pickle dict (`cache_{model}.pkl`).
Subset calls look up each hash individually; only misses trigger model inference.
The WT embedding for delta mode is stored in the same dict from `fit()`. Saves are
atomic via `tempfile.mkstemp` + `os.rename`.
